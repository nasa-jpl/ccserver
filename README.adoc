= CORECONF CoAP Server
:toc:
:toclevels: 3
:numbered:

A C++ implementation of a CORECONF (YANG-CBOR) server using libcoap, nanocbor, and ccoreconf.

== Overview

This server implements the CORECONF protocol for managing network devices using YANG data models over CoAP with CBOR encoding. It provides a complete implementation with auto-generated handlers from YANG models and SID files.

=== Supported Operations

The server implements the three main CORECONF operations:

* *GET (not yet implemented)*: Retrieve a single data node
* *FETCH*: Retrieve multiple data nodes in a single request
* *iPATCH*: Apply partial updates to configuration data

== Architecture

The server uses a layered architecture:

. *CoAP Layer*: Handles CoAP protocol communication (libcoap)
. *CORECONF Handler Layer*: Processes CORECONF requests and responses
. *SID Handler Registry*: Maps SIDs to read/write handlers
. *Implementation Layer*: User-provided business logic for data access
. *Datastore*: In-memory YANG data tree with CBOR serialization

=== Key Components

* *coreconf_handlers*: Main CORECONF protocol handlers (GET, FETCH, iPATCH)
* *sid_handlers*: Registry for SID-specific read/write handlers
* *ccoreconf*: YANG data model manipulation and CBOR serialization
* *Auto-generated handlers*: SID-specific handlers generated from YANG/SID files
* *Implementation stubs*: User-provided read/write functions for actual data access

== Dependencies

=== Required Libraries

* *libcoap-3*: CoAP protocol implementation
* *nanocbor*: Lightweight CBOR encoder/decoder (included as submodule)
* *ccoreconf*: CORECONF data model library (included as submodule)
* *cxxopts*: Command-line argument parsing (included as submodule)

== Building

[source,bash]
----
# Clone with submodules
git clone --recursive <repository-url>

# Or if already cloned, initialize submodules
git submodule update --init --recursive

# Build the library
mkdir build
cd build
cmake -DSID_FILES="../impl/simple-example@2026-06-04.sid" \
      -DINSTANCE_JSON="../impl/simple-example-instance.json" \
      -DGENERATED_NAME="simple-example" \
      ..
make

# Build the optional example executable
make example
----

The build process will:

. Build external dependencies (NanoCBOR, ccoreconf)
. Generate SID handlers from `.sid` files
. Generate model instance CBOR buffer
. Compile the static library (`libccserver.a`)

=== Build Targets

* `ccserver` (default) - Builds the static library for integration into other applications
* `example` - Builds the standalone example executable (not built by default)

The typical use case is to link against the `ccserver` library from your own application. The `example` target provides a reference implementation.

== Usage

=== Using as a Library

The primary use case is to link against `libccserver.a` in your own application:

[source,cmake]
----
# In your CMakeLists.txt
target_link_libraries(your_app PRIVATE ccserver)
----

See the `examples/main.cpp` for a reference implementation of how to initialize and run the server.

=== Running the Example Executable

[source,bash]
----
# Build the example first
make example

# Run with default settings (bind to all interfaces on port 5683)
./example

# Specify host and port
./example --host 127.0.0.1 --port 5684

# Show help
./example --help
----

=== Testing with CoAP Client

[source,bash]
----
# GET request
coap-client -m get coap://127.0.0.1:5683/c

# FETCH request (retrieve SID 1002)
echo "820403ea" | xxd -r -p | coap-client -m fetch -f - coap://127.0.0.1:5683/c

# iPATCH request (update SID 1002 to value 42)
echo "81a10403ea182a" | xxd -r -p | coap-client -m ipatch -f - coap://127.0.0.1:5683/c
----

== Implementation Guide

=== Adding Support for a New YANG Model

. *Add your YANG model* to the appropriate directory
. *Generate SID file* using pyang with SID plugin
. *Generate autocoded* handlers and implementation stubs
. *Implement read/write functions* by copying stubs into the `impl/` directory and modifying as needed
. *Update CMakeLists.txt* to include your impl files
. *Rebuild*

=== Handler Auto-Generation

The build system automatically generates SID-specific handlers from `.sid` files, as well as user implementation stubs.

[source,bash]
----
mkdir build
cd build
cmake -DSID_FILES=<your_sid_files.sid> -DINSTANCE_JSON=<your_model_instance.json> -DGENERATED_NAME=<your_base_name> ..
make generated_handlers
cp generated/<your_base_name>-impl-template.* ../impl
mv ../impl/<your_base_name>-impl-template.c ../impl/<your_base_name>-impl.c
mv ../impl/<your_base_name>-impl-template.h ../impl/<your_base_name>-impl.h
----

Thes auto-generated handlers:

. Extract keys from request context
. Call user-implemented read/write functions
. Update the in-memory datastore
. Handle type validation and error reporting

=== Updating Custom Implementation

After copying the implementation stubs, you can edit them as needed:

[source,c]
----
// Read function
uint32_t read_<module>_<identifier>() {
    // Return current value
    return current_value;
}

// Write function
int write_<module>_<identifier>(uint32_t new_value) {
    // Validate and store new value
    current_value = new_value;
    return 0; // Success
}
----

== CORECONF Protocol Details

=== FETCH Request Format

FETCH requests use CBOR arrays to specify which SIDs to retrieve:

----
[SID1, [SID2, KEY1, KEY2], SID3]
----

* Simple SID: Just the integer SID value
* SID with keys: Array where first element is SID, rest are key values

=== iPATCH Request Format

iPATCH requests use CBOR maps to specify SID-value pairs:

----
[{SID1: VALUE1}, {[SID2, KEY1]: VALUE2}]
----

* Simple updates: `{SID: value}`
* List item updates: `{[SID, key]: value}`

=== Response Format

Responses are CBOR-encoded data matching the request structure.

=== Common Issues
==== Memory Leaks

Because `ccoreconf` usese dynamic memory allocation, ensure proper cleanup of:

* `requestKeys` (DynamicLongListT)
* `requestElement` (CoreconfValueT)
* `requestValue` (CoreconfValueT) - only if not inserted into datastore

==== Garbage SID Values

Always check the Coreconf type (`CORECONF_UINT_32` vs `CORECONF_UINT_64`) before accessing `data.u32` or `data.u64` to avoid reading garbage from the wrong union member.
